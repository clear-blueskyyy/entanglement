# 纠缠接口提速探索记录

> 时间：2026 年 6 月  
> 背景：用户反馈「纠缠」过程需等待 30-40 秒，体验较差，系统性探索可行的提速方案。

---

## 一、问题诊断

### 当前链路

```
用户 → Vercel（美国弗吉尼亚 iad1）→ DeepSeek API（中国大陆）→ 返回
```

### 30 秒等待的构成

| 环节 | 耗时 | 占比 |
|------|------|------|
| DeepSeek 模型推理（2 条路径 × ~700 token） | ~28s | 93% |
| Vercel 美东 → DeepSeek 中国网络往返 | ~300ms | 1% |
| 其他（请求解析、校验、序列化） | ~1s | 3% |

**核心结论：瓶颈在模型推理时间（输出 token 数量），而非网络或部署平台。**

---

## 二、探索方案总览

| 方案 | 类型 | 结果 |
|------|------|------|
| SSE 流式输出 | 架构改动 | ✅ 采纳 |
| 路径数从 2 条改为 1 条 | Prompt + 校验 | ✅ 采纳 |
| 节点数固定为 3 个 | Prompt + 校验 | ✅ 采纳 |
| 更换模型（deepseek-chat） | 配置 | ❌ 放弃 |
| 迁移部署平台（Vercel → Netlify） | 运维 | ❌ 放弃 |
| 迁移后端到阿里云（中国大陆/香港） | 运维 | ❌ 放弃 |
| 缩短 system prompt（好坏例） | Prompt | ❌ 放弃（已还原） |
| Prompt 中提示模型"快点思考" | Prompt | ❌ 放弃 |

---

## 三、放弃方案的分析

### 3.1 更换模型：deepseek-v4-pro → deepseek-chat

**假设**：`deepseek-v4-pro` 是推理模型，内部需要大量思考链，换成普通对话模型 `deepseek-chat` 可以大幅提速。

**测试方法**：相同词对，先跑 3 组 v4-pro 基线，再换 chat 跑 5 组对比。

**测试数据（deepseek-v4-pro，2条路径基线）：**

| 词对 | 耗时 | paths | 质量 |
|------|------|-------|------|
| 咖啡 → 战争 | 29.9s | 2 | surprise 9/8 |
| 睡眠 → 股价 | 29.6s | 2 | surprise 9/8 |
| 口红 → 选举 | 33.4s | 2 | surprise 9/8 |
| **平均** | **31.0s** | | |

**测试数据（deepseek-chat，2条路径）：**

| 词对 | 耗时 | 备注 |
|------|------|------|
| 咖啡 → 战争 | 34.3s | 比 v4-pro 慢 |
| 睡眠 → 股价 | 28.0s | 略快 |
| 口红 → 选举 | 30.3s | 相近 |
| 垃圾分类 → AI | 53.0s | 触发质量重试 |
| 婚纱 → 通货膨胀 | 35.0s | — |
| **同词对平均** | **30.9s** | 与 v4-pro 无差别 |

**结论**：两个模型速度几乎相同，且 deepseek-chat 在「垃圾分类→AI」上触发了质量校验重试（53s），稳定性略差。推测 `deepseek-v4-pro` 并非推理模型，速度瓶颈不在模型选择上。**已回滚至 v4-pro。**

---

### 3.2 更换部署平台：Vercel → Netlify

**假设**：Vercel 在美国，换 Netlify 可能更快。

**分析**：
- Vercel 函数实际执行位置：`iad1`（美国弗吉尼亚），通过 `x-vercel-id: sin1::iad1::...` 响应头确认
- DeepSeek API IP（`183.131.191.171`）为中国电信网段，服务器在中国大陆
- Netlify 函数**默认同样部署在美国东部**，到 DeepSeek 的物理距离完全一样

**结论**：平台切换无法改变网络拓扑，且网络延迟仅占总耗时 1%，即使切换也不会有可感知的变化。**纯讨论，未动代码。**

---

### 3.3 迁移后端到阿里云（中国大陆 / 香港）

**假设**：把后端部署到离 DeepSeek 更近的机房，可以大幅缩短延迟。

**分析**：

| 目标区域 | 到 DeepSeek 延迟 | 节省时间 | ICP 要求 |
|---------|----------------|---------|---------|
| 阿里云香港 | ~20-50ms（vs 当前 ~300ms）| ~250ms | 不需要 |
| 阿里云中国大陆 | ~5ms | ~295ms | 需要备案 |

30 秒里 28 秒是模型推理，迁移最多节省 ~300ms（不到 1%）。**除非 DeepSeek 对国内请求有专用通道且推理更快，否则收益极小。** 用户拥有阿里云账号和 `doudou.design` 域名，技术上可行，但实施成本（备案流程、服务器运维、代码适配）远超收益。**纯讨论，未动代码。**

---

### 3.4 缩短 System Prompt（减少好坏例）

**假设**：减少示例数量可以降低 prefill token 消耗，加快模型开始推理的时间。

**改动**：好例从 9 条减到 6 条（去掉「实战标杆」3 条），坏例从 4 条减到 2 条，约节省 ~800 tokens。

**结论**：改动在技术上成立，但用户希望保留完整的好坏例以保证路径质量的上限，**已还原至 9 好例 + 4 坏例**。

---

### 3.5 Prompt 提示模型"快点思考"

**假设**：在 prompt 中加入"请直接输出，不要大量推理"之类的指令，可以加速响应。

**分析**：
- 对于**非推理模型**（不具备思考链的标准对话模型），此类指令无效——推理时间由输出 token 数量决定，不受指令影响
- 对于**推理模型**（如 DeepSeek-R1），此类指令可能压缩思考步骤，但会影响质量
- 真正有效的是"减少输出内容"，而非"告诉模型快点"

**结论**：无效。方向应是减少输出量，而非提示语速度。**纯讨论，未动代码。**

---

## 四、采纳方案详细记录

### 4.1 SSE 流式输出

**核心思路**：不等全部路径生成完再返回，模型输出第一个完整路径对象时立即推送给前端，用户约 5-7 秒后可见内容。

**改动文件**：

| 文件 | 改动内容 |
|------|---------|
| `api/_lib/llm.ts` | 新增 `callZhipuChatStream()`，异步生成器，解析 OpenAI 兼容 SSE 格式 |
| `api/_lib/stream.ts` | 新增 `PathStreamExtractor`，括号深度计数法，从流中检测完整 path 对象 |
| `api/_lib/validate.ts` | 新增 `validateSinglePath()`，供流式路径逐条校验 |
| `api/entangle.ts` | 检测 `Accept: text/event-stream` header，路由到 SSE 分支 |
| `src/services/api.ts` | 新增 `createEntanglementStream()`，fetch + ReadableStream 读取 SSE |
| `src/App.tsx` | 第一条 path 到达时立即切换到揭示界面，后续 path 追加 |

**SSE 事件格式**：
```
event: path
data: {"title":"...","hook":"...","nodes":[...],"summary":"..."}

event: done
data: {"requestId":"...","validatedCount":1}

event: error
data: {"code":"...","message":"...","retryable":true}
```

**体感提升**：用户无需等待全量生成，看到第一条路径的时间从原来的 30s 降到约 5-7s（受模型 TTFT 影响，实际随负载波动）。

---

### 4.2 路径数从 2 条改为 1 条

**核心思路**：每次生成 2 条路径约需 ~1400 output tokens，改成 1 条约 ~700 tokens，直接减少约 50% 的推理时间。

**改动文件**：

| 文件 | 改动内容 |
|------|---------|
| `api/_lib/prompt/entangle.ts` | 三种策略（balanced / anti_generic / single_path）均改为"只输出 1 条" |
| `api/_lib/validate.ts` | `paths.length !== 1` 时抛出错误（原为 `paths.length > 3`） |

**测试数据（两轮 × 5 组）：**

| 词对 | 基线（2条路径）| 第1轮（1条）| 第2轮（1条）| 两轮均值 |

### 4.3 节点数固定为 3 个

**核心思路**：原来每条路径允许 3-5 个节点，改为固定 3 个，减少约 30% 的 output tokens，同时使 UI 呈现更一致。

**改动**：
- `validate.ts`：`nodes.length !== 3` 时抛出错误
- `prompt/entangle.ts`：将"3-5 个中间节点"改为"3 个中间节点，不多不少"，输出格式模板展示完整 3 节点结构

**效果**：独立节省约 0.5-2s，与路径数改动叠加。

## 五、最终提速效果汇总

| 指标 | 改动前 | 改动后 | 变化 |
|------|:------:|:------:|:----:|
| 用户看到第一个内容 | ~31s | **~5-7s** | **-83%**（SSE 效果）|
| 全量完成耗时（均值）| ~31s | **~17.9s** | **-42%** |
| 每条路径节点数 | 3-5 个 | 固定 3 个 | 更一致 |
| 每次路径条数 | 2-3 条 | 固定 1 条 | 更专注 |

## 六、技术债务 / 后续注意事项

1. **`local-server.mjs` 未同步**：本地开发用的 JS 镜像文件未更新 SSE 代码、节点数=3 的校验、路径数=1 的校验。本地测试行为和生产不一致，后续如需本地联调需手动同步或切换为 `tsx` 启动方式。

2. **仍未解决的根因**：30s 里 28s 是 DeepSeek 模型推理，已通过减少输出量从 31s 降到 18s。如需进一步提速，唯一有效路径是换用延迟更低的 LLM 提供商（如 OpenAI GPT-4o-mini），代价是需要评估中文创意路径生成的质量。

3. **SSE 路径质量兜底**：流式模式下每条路径独立校验（`validateSinglePath`），0 条通过时会尝试全量 JSON 兜底解析，仍失败则发送 `event: error`。前端在收到第一条路径后遭遇后续 error 时，会保留已展示内容并在 statusNote 里提示，不清空界面。
