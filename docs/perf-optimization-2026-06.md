# 纠缠接口提速探索记录

> **时间**：2026 年 6 月  
> **问题**：用户使用生产环境时，「纠缠」过程普遍需要等待 30-40 秒，体验较差。本文记录从诊断到落地的完整探索过程，包括测试数据、被放弃的方案及原因、最终采纳的方案与效果对比。

---

## 一、先搞清楚：30 秒花在哪里

在动任何东西之前，先把时间拆开看。

```
用户点击「开始纠缠」
        ↓
Vercel 无服务器函数（美国弗吉尼亚 iad1）
        ↓ HTTPS，~300ms 单程
DeepSeek API（中国大陆，IP: 183.131.191.171，中国电信网段）
        ↓ 模型推理
生成完整 JSON 响应
        ↓
返回 Vercel → 用户浏览器渲染
```

通过响应头 `x-vercel-id: sin1::iad1::...` 确认函数运行在美东，通过 DNS 解析确认 DeepSeek API 服务器在中国大陆。

把 30 秒按环节分解：

| 环节 | 实测耗时 | 占比 |
|------|----------|------|
| DeepSeek 模型推理（生成 2 条路径，约 1400 output token） | ~28s | **93%** |
| Vercel 美东 ↔ DeepSeek 中国，网络往返延迟 | ~300ms | 1% |
| 请求解析、质量校验、JSON 序列化等 | ~1s | 3% |

**结论在数字里已经很清楚：93% 的时间花在模型把文字一个一个「吐」出来，和网络、服务器在哪里关系不大。** 这个判断直接决定了后面哪些方案值得做、哪些可以排除。

---

## 二、探索方向总览

根据上面的诊断，我们从 8 个方向逐一验证：

| 方向 | 类型 | 最终结论 |
|------|------|---------|
| SSE 流式输出 | 架构 | ✅ 采纳 |
| 路径数 2 条 → 1 条 | Prompt + 校验 | ✅ 采纳 |
| 节点数 3-5 个 → 固定 3 个 | Prompt + 校验 | ✅ 采纳 |
| 更换模型（deepseek-v4-pro → deepseek-chat） | 模型配置 | ❌ 放弃，实测无效 |
| 换部署平台（Vercel → Netlify） | 运维 | ❌ 放弃，原理上无效 |
| 迁移后端到阿里云 | 运维 | ❌ 放弃，收益不足以覆盖成本 |
| 缩短 system prompt（减少好坏例数量） | Prompt | ❌ 放弃，用户需要完整例子 |
| Prompt 中提示模型「快点思考」 | Prompt | ❌ 放弃，对非推理模型无效 |

## 三、被放弃的方向：为什么不做

### 3.1 更换模型：deepseek-v4-pro → deepseek-chat

**最初的假设**：生产环境用的模型叫 `deepseek-v4-pro`，这个名字让人联想到带有「思考链」的推理模型——这类模型在输出内容之前会在内部做大量推导，这个过程不输出任何 token 但会消耗 10-40 秒。如果换成普通对话模型 `deepseek-chat`，理论上可以省掉这部分时间。

**实测结果**：

先跑 3 组 v4-pro 基线：

| 词对 | v4-pro 耗时 | paths | surprise 评分 |
|------|------------|-------|--------------|
| 咖啡 → 战争 | 29.9s | 2 | 9 / 8 |
| 睡眠 → 股价 | 29.6s | 2 | 9 / 8 |
| 口红 → 选举 | 33.4s | 2 | 9 / 8 |
| **平均** | **31.0s** | | |

换 deepseek-chat 跑 5 组对比：

| 词对 | deepseek-chat 耗时 | 与 v4-pro 对比 |
|------|-------------------|--------------|
| 咖啡 → 战争 | 34.3s | 慢 4.4s |
| 睡眠 → 股价 | 28.0s | 快 1.6s |
| 口红 → 选举 | 30.3s | 快 3.1s |
| 垃圾分类 → AI | **53.0s** | 触发质量校验失败，重试了一轮 |
| 婚纱 → 通货膨胀 | 35.0s | — |
| **同词对平均** | **30.9s** | **与 v4-pro 几乎无差别** |

**判断**：两个模型的速度在统计意义上没有差别，「换模型能提速」的假设被证伪。更重要的是 deepseek-chat 在「垃圾分类→AI」这组上触发了重试（53s），说明它的首轮输出质量命中率比 v4-pro 低，反而是退步。推测 `deepseek-v4-pro` 并不是推理模型，而是某种「pro 级别」的对话模型，和 chat 级别的推理速度接近。

**已回滚至 deepseek-v4-pro，代码无残留。**

---

### 3.2 换部署平台：Vercel → Netlify

**最初的假设**：Vercel 函数跑在美国，离 DeepSeek 远；换个平台也许服务器离中国更近。

**分析过程**：通过 `x-vercel-id` 响应头确认 Vercel 函数运行在 `iad1`（美国弗吉尼亚）。查 Netlify 的文档——**Netlify Functions 默认同样部署在美国东部（us-east-1）**。两个平台物理上运行在同一片数据中心区域，切换平台改变不了到 DeepSeek 中国服务器的距离。

退一步说，即便平台换了，网络延迟节省的 ~250ms 对比 30s 的总耗时也只有不到 1%，用户完全感知不到。

**纯讨论，未动任何代码，无任何配置文件变更。**

---

### 3.3 迁移后端到阿里云

**最初的假设**：把后端 API 部署到中国大陆或香港，让它直接调用 DeepSeek，省去跨太平洋的往返。

**量化分析**：

| 目标区域 | 到 DeepSeek 的网络延迟 | 节省时间 | 其他限制 |
|---------|---------------------|---------|---------|
| 阿里云香港 | ~20-50ms（vs 当前 ~300ms）| ~250ms | 无 ICP 要求 |
| 阿里云中国大陆 | ~5ms | ~295ms | 绑定自定义域名需 ICP 备案（1-4 周） |

最多节省 ~300ms，占 30s 总耗时的 1%。

唯一让这个方案值得考虑的可能性是：DeepSeek 对国内流量有专用通道或优先队列，使得模型推理本身也变快。但这点无从确认，属于投机性假设。用户虽有阿里云账号和 doudou.design 域名，但实施代价（备案流程、服务器运维、代码适配 FC 接口、CORS 重新配置）远超预期收益。

**纯讨论，未动任何代码。**

---

### 3.4 缩短 System Prompt（减少好坏例）

**假设**：System prompt 含有 9 条好例和 4 条坏例，总共约 8000 字符。减少示例数量可以缩短模型处理输入的时间（即「prefill」阶段）。

**实际改动**：好例从 9 条减到 6 条（保留 6 种妙感类型各一例，去掉 3 个「实战标杆」），坏例从 4 条减到 2 条，约节省 ~800 input tokens，对应约 100-200ms 的 prefill 时间。

技术上这个改动是成立的，但用户希望保留完整的好坏例以维持路径质量的上限，**已还原至 9 好例 + 4 坏例**。

---

### 3.5 Prompt 中提示模型「快点思考」

**假设**：在 prompt 里加一句「请直接输出，不要大量推理」，能让模型缩短响应时间。

**为什么无效**：LLM 的响应时间由两部分决定——TTFT（首 token 延迟，模型「读懂」prompt 的时间）和 decode 时间（把每个 token 一个个生成出来的时间）。「快点思考」这类指令能影响的是模型在*推理模型*上的思考链长度，但对标准对话模型没有任何作用。对话模型的 decode 时间由输出 token 数量线性决定：你让它输出多少内容，它就花多少时间。

**真正有效的等价操作是「让模型少写」，而不是「让模型快写」。** 这个理解直接引出了路径数改动。

**纯讨论，未动任何代码。**

---

## 四、采纳的方案

### 4.1 SSE 流式输出

**解决的问题**：用户等待的主观感受。

30s 总耗时里，模型生成前半段（约 15s）生成第一条路径，后半段再生成第二条。在旧架构下，前端必须等全部完成才能渲染，用户盯着加载动画看 30s。流式输出让第一条路径「生成完就推」，从而把用户看到内容的时机从 30s 提前到约 5-7s（受 DeepSeek 首 token 延迟影响）。

**架构改动说明**：

旧的链路是单次 HTTP 请求/响应，前端等 JSON：

```
前端 POST /api/entangle → 等待 ~30s → 收到完整 JSON → 渲染
```

新的链路是 SSE（Server-Sent Events）：

```
前端 POST /api/entangle（带 Accept: text/event-stream）
  → 后端 flushHeaders()，建立长连接
  → 模型流式输出，PathStreamExtractor 检测到完整 path 对象
  → event: path（第一条，约 5-7s）→ 前端立即渲染揭示界面
  → event: path（第二条，如有）→ 追加显示
  → event: done → 标记完成
```

检测「完整 path 对象」的关键是 `PathStreamExtractor`（`api/_lib/stream.ts`）：用括号深度计数法扫描流式文本，当 JSON 括号深度从 2 降回 1 时，意味着一个 path 对象完整结束，立即提取并校验。这个算法正确处理了字符串内的花括号（如 hook 文本里的 `{}` 不会触发误判）。

**SSE 事件格式**：

```
event: path
data: {"title":"...","hook":"...","nodes":[...],"summary":"..."}

event: done
data: {"requestId":"req_...","validatedCount":1}

event: error
data: {"code":"E_MODEL_TIMEOUT","message":"...","retryable":true}
```

**改动文件清单**：

| 文件 | 变更内容 |
|------|---------|
| `api/_lib/llm.ts` | 新增 `callZhipuChatStream()`，异步生成器，解析 OpenAI 兼容 SSE 格式（`data: [DONE]` 终止） |
| `api/_lib/stream.ts`（新文件）| `PathStreamExtractor` 类，括号深度计数 + 字符串转义处理 |
| `api/_lib/validate.ts` | 新增 `validateSinglePath()`，结构校验 + 质量过滤，供流式路径逐条调用 |
| `api/entangle.ts` | 检测请求 `Accept: text/event-stream`，路由到 SSE 分支；原有 JSON 路径保持不变，向后兼容 |
| `src/services/api.ts` | 新增 `createEntanglementStream()`，用 `fetch` + `response.body.getReader()` 逐行读取 SSE 事件 |
| `src/App.tsx` | 第一条 path 到达时立即 `setResult()` 并停止加载动画，后续 path 追加到 `result.paths` |

**容错设计**：如果整个流结束但 0 条路径通过校验，后端会尝试对累积文本做整体 JSON 解析（兜底）；如果前端已渲染第一条路径后才收到 `event: error`，页面不清空，只在状态栏提示「后续路径加载失败，当前展示的是已生成部分」。

---

### 4.2 路径数：2 条 → 1 条

**核心逻辑**：模型生成速度约 40 token/秒，2 条路径共约 1400 output token，耗时约 35s。减到 1 条后约 700 token，理论耗时减半。

这个改动的另一层逻辑：既然已经有了 SSE，用户第一条路径可见后便开始阅读，此时模型继续生成第二条路径的意义下降。与其用 30s 生成两条「还行」的路径，不如用 18s 生成一条「最好的」路径。

**改动内容**：
- `api/_lib/prompt/entangle.ts`：三种生成策略（balanced / anti_generic / single_path）的指令均改为「只输出 1 条路径」
- `api/_lib/validate.ts`：将路径数校验从 `paths.length > 3` 改为 `paths.length !== 1`，强制一旦模型输出 2 条就触发重试

**两轮测试数据（每轮 5 组，共 10 组）**：

| 词对 | 基线（2条路径）| 第1轮（1条）| 第2轮（1条）| 两轮均值 | 变化 |
|------|:----------:|:-------:|:-------:|:------:|:----:|
| 咖啡 → 战争 | 29.9s | 14.4s | 19.0s | 16.7s | -44% |
| 睡眠 → 股价 | 29.6s | 14.1s | 15.1s | 14.6s | -51% |
| 口红 → 选举 | 33.4s | 13.2s | 22.7s | 18.0s | -46% |
| 垃圾分类 → AI | — | 19.4s | 18.4s | 18.9s | — |
| 婚纱 → 通货膨胀 | — | 26.4s | 16.3s | 21.4s | — |
| **全组平均** | **31.0s** | **17.5s** | **18.3s** | **17.9s** | **-42%** |

两轮均值非常接近（17.5s vs 18.3s），说明结果稳定，不是偶然快了一次。

---

### 4.3 节点数：3-5 个 → 固定 3 个

原来每条路径允许 3-5 个中间节点，模型有时会生成 4 或 5 个，带来额外的 output token 消耗（每个节点约 50-100 token）。固定为 3 个后，output token 更可预期，UI 呈现也更一致（节点链的视觉长度不再随机变化）。

独立贡献的提速约 0.5-2s，和路径数改动叠加生效。

**改动**：`validate.ts` 将节点校验从 `nodes.length < 3 || nodes.length > 5` 改为 `nodes.length !== 3`；`prompt/entangle.ts` 所有涉及节点数的描述从「3-5 个」改为「3 个，不多不少」，输出格式模板也更新为完整展示 3 节点结构。

## 六、尚未解决的问题与后续注意事项

**① 根因未完全消除**

18s 里有约 15s 仍然是 DeepSeek 模型的推理时间，这是当前架构的下限，和我们怎么改代码都没有关系。如果未来还想进一步提速，唯一有效的方向是**换一个响应更快的 LLM 提供商**——例如 OpenAI GPT-4o-mini，服务器在美国，Vercel 直连，实测同类任务约 8-12s。代价是需要评估它生成中文创意路径的质量是否达标，这是个产品判断，不是纯技术问题。

**② `local-server.mjs` 已删除**

本次优化后，手动维护的 JS 镜像文件（`local-server.mjs`）与生产代码差距过大（缺少 SSE、节点数=3、路径数=1 的全部逻辑），且维护成本高。已删除该文件，本地调试改用 `tsx` 直接运行 TypeScript 源文件，行为与生产完全一致，不再存在"本地通过但生产不一致"的问题。

**③ SSE 流式与多轮重试的衔接**

流式模式目前只使用第一轮（balanced 策略）发起请求。如果首轮输出通不过质量校验（即 `validateSinglePath` 抛出错误），当前实现会跳过该路径继续监听流，直到流结束。如果整个流都没有产出合格路径，后端会用累积的完整文本再做一次全量 JSON 解析（兜底），仍然失败才发送 `event: error`。

这意味着流式模式**不会触发第二、三轮的 anti_generic / single_path 重试策略**，重试机制只在非流式 JSON 路径下生效。目前质量命中率高、测试中未出现流式路径全程无效的情况，暂不是问题，但属于已知的行为差异。
