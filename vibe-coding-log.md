# 纠缠 — Vibe Coding 过程记录

> 这份文档记录整个项目的决策过程、走过的岔路、选择的理由和经验教训。
> 供下次 Vibe Coding 项目直接参考，不是给用户看的，是给自己和 AI PM 看的。

---

## 一、项目启动阶段

### 1.1 项目定位的确定

**初始状态**：有两份文档（项目文档 + 技术交付文档），内容重叠、技术与产品定义混在一起，AI Agent 读完不知道该干什么。

**走过的岔路**：
- 第一版项目文档写成了散文腔，大量"它应该带一点安静夜晚的感觉"这类表达，对 Agent 无指导价值
- 技术交付文档里夹带了产品立意（"为什么值得做"等），造成两份文档职责不清

**最终决策**：
- `项目文档.md`：只回答"做什么、做到什么程度、什么算对了、什么不做"
- `技术交付文档.md`：只回答"怎么实现、怎么验收、怎么部署"
- 两份文档职责分离，互不越界

**经验教训**：
> 给 AI Agent 的文档要"可被执行"，而不是"可被欣赏"。每一句话都应该能回答"Agent 读完这句话，下一步会做什么不同"。如果一句话删掉对 Agent 行为没影响，就可以删。

---

### 1.2 技术方案选型的确定

**核心分歧**：两份原始文档的 AI 模型选择不一致，一个选智谱 GLM，一个选 DeepSeek。

**分析过程**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 智谱 GLM-4-Flash | 永久免费，无过期压力，国内低延迟 | 鉴权方式特殊（需生成 JWT Token） |
| DeepSeek | 推理能力强，送额度 | 额度有限，用完需付费 |

**最终决策**：主力用智谱 GLM-4-Flash，扣子工作流作降级备选。理由：MVP 阶段要反复调试 prompt，零成本比推理质量更重要；鉴权复杂的问题放在 Serverless 函数里处理，前端不感知。

**经验教训**：
> MVP 阶段首选零成本方案，不要为了"推理质量可能更好"而引入有限额度的模型。调 prompt 需要大量调用，算力成本应该接近零。质量不够再换模型，而不是一开始就用最贵的。

---

### 1.3 架构的确定：Serverless 代理 vs 纯前端

**初始方案**：技术交付文档原始版本写的是"纯前端部署，所有逻辑在浏览器端完成"。

**问题发现**：
- 纯前端直连模型 API，API Key 暴露在浏览器代码中
- 产品打算公开分享链接，任何人都能从 DevTools 拿走 Key 刷额度
- 即使是永久免费的智谱 GLM，Key 被滥用也会有调用频率压力

**最终决策**：强制要求 Serverless API 代理层（Vercel Functions），API Key 只存服务端环境变量，前端不持有任何凭证。

**经验教训**：
> "个人项目可以先不管安全"这个说法在公开分享场景下不成立。一旦链接发出去，就要假设有人会找 Key。Serverless 代理不增加成本，只多 30 分钟，没理由不做。

## 二、产品定义阶段

### 2.1 关于"每日配对"的内容真实性

**初始设定**：用 AI 生成"今日热点"作为配对词来源。

**问题发现**：AI 生成热点有编造风险，会产生不存在的新闻或假话题。

**关键判断**：
> AI 在这个产品里的作用是"发现连接"，不是"编造事实"。配对的词必须是真实可查的，关联路径可以是出人意料的。

**最终决策**：配对词必须真实存在。AI 生成时加强 prompt 约束（"严禁编造不存在的内容"），不追求实时热搜，真实+有趣优于新鲜+虚假。

**经验教训**：
> AI 产品设计时要想清楚 AI 在哪个环节发挥作用、不该在哪个环节发挥作用。边界模糊会导致产品质量不可控。

---

### 2.2 每日配对的一致性：每次随机 vs 同一天固定

**分歧来源**：技术文档里写"适度随机"，但功能清单有"换一组"按钮，两者逻辑矛盾。

**分析**：
- 方案A（同一天固定）：增加"今日"仪式感，但如果只用一天，变化感弱
- 方案B（每次随机）：使用场景集中在一天内反复体验，每次打开刷新，"换一组"也随机

**最终决策**：方案B，每次随机。理由：使用场景是集中一天内玩，需要更高刷新频率；"每日"是产品标签，不是技术约束。

**经验教训**：
> 功能描述的"适度随机"是模糊表达，在文档中容易造成误解。所有涉及"随机/固定/每日"的行为都要明确到：触发条件是什么、每次打开的结果是否一致。

---

### 2.3 分享卡片的优先级

**初始定位**：分享卡片放在阶段4（最后阶段）。

**重新评估**：产品的唯一传播路径就是分享卡片，它不是锦上添花，而是"产品能不能被传播"的关键。

**最终决策**：分享卡片保留为 MVP 功能（✅），与路径展开、历史记录同等优先级。

**经验教训**：
> 单次体验闭环的产品，传播机制不是运营策略，而是产品功能本身。要早做、做好，不要留到最后"打磨"时才做。

## 三、文档管理阶段

### 3.1 Agent 模式下的文档规范

**问题**：早期文档写法接近"产品设计思考记录"，对 Agent 的指导价值有限。

**教训清单**：
1. **避免散文腔**：形容性语言（"带一点安静夜晚的感觉"）对 Agent 无效，换成约束性描述（"深色背景，非纯黑，带深蓝或深灰调"）
2. **每条规则要可被验收**：无法回答"做到了吗"的验收标准没有价值
3. **不做什么和做什么同等重要**：明确的"不做"列表能防止 Agent 过度发挥
4. **优先级原则要显式写出**：当 Agent 遇到取舍时，靠猜比靠原则更容易出错

### 3.2 两份文档的职责边界

| 文档 | 回答的问题 | 不应该包含 |
|------|-----------|-----------|
| `项目文档.md` | 做什么、做到什么程度、什么算对了、什么不做 | 技术实现、接口设计、框架选型 |
| `技术交付文档.md` | 怎么实现、怎么验收、接口契约、部署规范 | 产品立意、运营策略、用户分析 |

**经验教训**：
> 两份文档并存时，职责不清会导致 Agent 在两份文档中找到矛盾的指令。建议在每份文档开头明确"本文档不包含什么"。

### 4.3 设计重置记录（2026-05-05）

**触发背景**：项目在技术上已经基本可跑，但用户预览后明确反馈：当前版本"毫无设计，像输入框和文本的堆砌"，而且请求超时进一步放大了这种负面感受。问题不再是"功能有没有"，而是"产品有没有被呈现成一个作品"。

**关键转向**：
- 暂停继续纠缠技术选型、模型质量和 Prompt 微调，先回到产品呈现与 UI/UX 设计
- 接受一个现实：当用户说"这不像产品经理做出来的东西"时，优先级最高的是页面叙事、视觉语言和交互节奏，而不是继续补局部功能
- 明确设计讨论应采用"问题拆解 + 选项对齐"方式，而不是泛泛地问"你想要什么感觉"

**本轮对齐出的设计锚点**：
- 产品定位：**碎片时间的智性消遣**
- 用户情绪：**好奇 + 挑逗**
- 视觉参考：**Nautilus 杂志**（深色、学术感、克制、优雅）
- 首页结构：**产品名称（中英文）+ 标语 + 两个独立输入框 + CTA + 今日推荐标签**
- 结果体验：**逐节点揭示**，每次点击揭开一步，中间有粒子流动，结尾有收尾仪式
- 页面过渡：**首页收缩上移，结果在下方展开**，保留单页连续感
- 背景隐喻：**太空星网**，散落星点、细连线、粒子流动，服务于"暗线"概念，而不是单纯装饰
- 品牌表达：`ENTANGLEMENT / 纠缠`，标语为 **"万物皆有暗线"**

**额外补齐的流程决策**：
- 设计讨论结束后，必须立即沉淀成设计文档，否则实现阶段很容易凭感觉漂移
- 为了让设计评审不被接口稳定性拖住，需要提供一版可直接预览的静态体验（哪怕先用示例数据）
- 产品文档、设计规范、技术文档三者要继续分层：产品定义目标，设计文档定义呈现，技术文档定义实现

**经验教训**：
> 用户吐槽"效果很差"时，不要本能地继续调 Prompt。很多时候，用户批评的不是单点能力，而是整个产品没有被组织成一个完整体验。此时应先回答：第一页让人想不想点、展开过程有没有悬念、最后有没有记忆点。把这些解决了，再谈模型质量，效率更高。

### 4.4 模型质量方向纠偏（2026-05-07）

**触发背景**：在本地可跑、UI 重置、分享链路补齐之后，用户用"阿黛尔 → AI Agent"做真实测试，明确反馈生成结果"牛头不对马嘴"、"没有深度"、"一眼看出是保底路径"。这说明当前版本的问题已经不再是页面像不像产品，而是产品最核心的一步——"纠缠路径"本身没有成立。

**这次翻车的本质，不是技术没补完，而是优先级判断错了**：
- 用户真正购买的不是"有一条路径"，而是"这条路径居然真能这样连上"的瞬间
- 一旦主生成链路长期失败，页面再完整、动效再漂亮、分享卡片再精致，都只是把一个空心产品包装得更像产品
- 保底机制本来应该只负责"不要空白"，但这次实际承担了主流程内容供给，说明系统已经在错误层级上维持可用

**这次具体错在哪里**：
1. **把"页面不空白"误当成"核心功能可用"**  
   当接口能稳定返回一份结构完整的结果时，很容易产生"功能至少跑通了"的错觉。但对这类生成式产品来说，真正的可用标准不是有没有结果，而是结果能不能让人信、能不能让人停留、能不能让人愿意分享。
2. **把"用户嫌内容差"误解成"文案包装不够好"**  
   用户已经明确指出"概念之间没有相关度"，这不是文案修饰问题，而是产品价值层的失败。此时继续优化保底标题、hook、summary，本质是在掩盖主链路失败。
3. **把"保底更自然"放到了比"主链路成功率"更靠前的位置**  
   这会形成很危险的开发惯性：因为 fallback 容易改、见效快，就不断在 fallback 上打补丁；而真正难的模型链路因为见效慢，反而被延后。结果是系统越来越会"体面地失败"，但越来越不会"真正地成功"。
4. **没有建立生成式产品专属的验收标准**  
   之前的验收更偏工程：能 build、能返回 JSON、页面能展示。但这类产品真正需要的是内容验收：相关度够不够、节点具体不具体、有没有意外感、是否像人工硬凑的模板。

**以后再做类似产品时，优先级必须这样排**：
1. **先证明核心惊喜成立**：先验证模型能不能稳定产出 5-10 组让人点头的结果，再去做视觉、缓存、分享
2. **再保证弱网和失败态不难看**：fallback 只在主链路大致成立后再打磨
3. **最后才补传播闭环和外围体验**：分享、历史、花哨动效都排在核心生成质量之后

**可直接复用的防翻车规则**：
- 如果产品的价值主要来自生成结果本身，第一阶段的主 KPI 不是 DAU、不是 UI 完整度，而是**首批样本通过率**
- 在做任何"体验打磨"前，先拿至少 10 组跨度大的真实输入做人工评测，至少回答四个问题：
  - 这条路径是不是明显不是模板拼接？
  - 中间节点是不是具体世界对象，而不是抽象大词？
  - 每一跳是不是都能说清为什么能到下一跳？
  - 用户看到后会不会说"原来还能这样连"？
- 如果连续 3 组真实测试都被用户一眼识别为保底或模板，就必须暂停继续做 UI/外围功能，回到模型层排查
- 文档里要显式写清楚：`fallback` 是**可用性策略**，不是**体验策略**；触发率高说明主功能失效，不是系统健壮

**后续做 Vibe Coding 时可照抄的检查顺序**：
- 第一步：先写出产品的"Aha Moment"到底是什么，用户愿意截图/分享的是哪一刻
- 第二步：把所有任务按"是否直接提升这个 Aha Moment"排序
- 第三步：如果某个改动只能让失败更体面，但不能提高 Aha Moment 的出现概率，就降级优先级
- 第四步：每轮迭代结束后，不只看"有没有功能新增"，还要看"核心惊喜的命中率有没有上升"

**经验教训**：
> 当 AI 产品的核心价值直接落在生成质量上时，不能因为系统需要兜底，就把优化重心放到兜底机制上。兜底只能保证"不空白"，不能证明"产品成立"。一旦用户反馈集中在"内容不对"，就应该立即停止润色保底，转去解决主生成链路。
> 
> 更具体一点：以后只要看到自己在反复优化 fallback 标题、fallback 文案、fallback 过渡，而没有同步提升主链路命中率，就要立刻警觉——这通常意味着方向已经偏了。

### 4.5 这次翻车暴露出的判断偏差（2026-05-07）

**先记结论**：这次不是"模型偶尔不稳定"，而是**产品定义、模型选型、提示词设计、验收方式**四层一起偏了，最后被 fallback 掩盖。

**偏差1：把"免费可反复调"当成了长期正确解**  
- 在搭 MVP 骨架时，`glm-4-flash` 的价值很大：零成本、低门槛、能让链路先跑起来
- 但到了要验证"这条路径本身值不值得用户停下来"时，评估标准已经变了：此时最重要的是**认知质量**，不是**调用成本**
- 继续坚持免费模型，会出现一个很危险的现象：开发成本很低，但产品价值长期不成立，最后省下的是 token，浪费的是整个产品判断周期

**可直接复用的判断规则**：
- 如果任务同时要求**跨域联想 + 因果解释 + 结构化输出**，就不要默认免费轻模型能扛住
- 如果一个功能的价值直接来自内容质量，而不是来自调用成功本身，模型成本就不该排在能力前面
- 免费模型适合：MVP 骨架、接口联调、前后端串通、错误结构调试
- 强模型适合：正式内容生成、首批用户验证、样本质量评测

**偏差2：把"有结果返回"误判成"用户已经感受到产品价值"**  
当页面上出现一条看起来完整的路径时，人很容易松一口气，觉得至少结果出来了。但用户真正买单的是：
- 这条路径是不是让我停下来看完
- 我是不是愿意点开每一跳
- 我看完会不会觉得"靠，这个居然真的连上了"

如果这些都没有发生，那么返回再完整，也只是**结构完整，不是体验成立**。

**以后必须分开看两套成功标准**：
- **工程成功**：接口通、JSON 合法、页面不报错、状态清晰
- **产品成功**：用户觉得相关、具体、有转折、有传播欲

只满足前者，绝不能宣布这个功能已经成立。

**偏差3：太早进入"体验打磨心态"，太晚回到"核心价值验证"**  
这次有一段时间，优化动作主要集中在：
- fallback 的标题更不像 fallback
- summary 更顺口
- 页面等待态和展示态更完整
- 缓存、提示、收尾体验更自然

这些事情并不是错的，但它们都建立在一个前提上：**主内容大致成立**。如果这个前提不存在，所有体验打磨都会变成包装失败。

**以后做类似项目时，必须给自己加一条硬规则**：
> 当用户第一次明确指出"内容不对"，后续一个迭代周期内，至少 70% 的精力必须投入主生成链路；除非该改动能直接提升主链路质量，否则一律降级。

**偏差4：没有提前写出生成式产品专属的验收题**  
之前更像在验一个普通前后端项目：能 build、能请求、能展示、能分享。但生成式产品必须在开发初期就写出自己的人工评测题。

**以后每做一个生成式产品，至少先写这 8 个问题**：
1. 用户的 Aha Moment 是哪一秒？
2. 什么叫一条"明显不行"的结果？请提前写出 3 个 bad case
3. 什么叫一条"值得截图"的结果？请提前写出 3 个 good case
4. 连续测 10 组真实输入时，至少有几组必须达到可分享水平，才能算产品成立？
5. 如果 fallback 触发率很高，团队会不会被"至少还有结果"麻痹？
6. 如果用户说"牛头不对马嘴"，第一优先排查的是模型、prompt、校验、还是 UI？
7. 哪些改动只是在让失败更体面？
8. 当前迭代里，哪项工作最直接提升 Aha Moment 的出现概率？

这些问题不写，后面就很容易用工程进度代替产品进度。

### 4.6 后续做 Vibe Coding 产品的防翻车清单（可直接照抄）

**一、先判任务负载，再定模型，不要默认用最省钱的**
- 如果任务只要分类、抽取、改写、轻总结，免费模型通常够用
- 如果任务要求"先理解隐含场景，再做多跳推理，再输出严格结构"，就要默认这是中高认知负载任务
- 中高认知负载任务一开始就应准备两档模型：`开发调试模型` 和 `正式生成模型`
- 不要等用户骂了再补一句"也许该升模型"；模型能力是产品方案的一部分，不是实现细节

**二、把 prompt 当成产品机制设计，不要当成长文案堆砌**
- prompt 不是把要求写得越多越好，而是要让模型知道：先做什么、再做什么、什么绝对不允许做
- 好 prompt 至少要分四层：角色定位、任务拆解、正反例、输出格式
- 如果一个 prompt 同时塞进产品立意、十几条硬规则、一个孤例、再加长 JSON schema，弱模型极容易只记住"输出 JSON"，忘掉真正重要的内容质量约束
- Prompt 不是越长越强，**能被模型稳定执行**才是强

**三、先设计人工评测，再做自动校验**
- 自动校验只能保证结构、字段、基础黑名单，不会替你判断"这条路径妙不妙"
- 在校验规则落地前，先做一版人工评测表：相关度、具体度、因果感、反常识转折、可传播性
- 如果人工一眼看出"模板味很重"，说明问题在生成策略，不在校验器
- 校验器的职责是拦掉明显坏结果，不是替代内容评审

**四、把 fallback 明确定义为报警器，而不是创可贴**
- fallback 一旦频繁触发，应该被视为红灯，而不是系统健壮
- 只要发现自己开始投入大量时间优化 fallback 的自然度，就要反问：是不是主链路已经坏了
- 对生成式产品来说，最危险的状态不是报错，而是**体面地输出没价值的内容**

**五、每轮迭代复盘时，优先看这三件事**
- 主链路真实产出率有没有上升
- 用户看到结果后的第一反应有没有变好
- 当前新增工作是不是更接近 Aha Moment，而不是更远

**经验教训**：
> 以后做 Vibe Coding，不要再把"能跑起来"误当成"方向没问题"。能跑只说明工程在工作，不能说明产品在成立。凡是核心价值由生成内容承载的项目，第一优先级永远是验证：模型能不能持续产出让人信服、愿意停留、愿意分享的结果。

### 4.7 模型提供方切换与真实可用性验证（2026-05-07）

**触发背景**：在明确“主链路质量是 P0”之后，下一步不再是继续修 fallback，而是验证：更强模型到底能不能接上、能不能稳定用、以及接上之后是否真的带来产品层改观。

**本轮实际发生的事情**：
1. **先尝试直接接 OpenAI 官方 API**  
   - 工程上很快就接通了，说明当前项目改造成“多提供方可切换”并不难
   - 但真实调用时遇到 `insufficient_quota`，证明“技术上可接入”不等于“业务上可持续可用”
   - 这个阶段的重要价值，不是拿到了可用模型，而是把调用层从“只支持智谱”升级成了“可切换模型底座”
2. **再切内部 FRIDAY One-API + App ID**  
   - 这一步把问题从“外部账号额度”切到了“公司内部统一模型平台能力与配额”
   - `gpt-5.4` 能识别、请求能到达上游，但立刻撞上 RPM 限流，说明平台路径是通的，只是当前 App ID 对该模型的调用配额不够
3. **再降一档切到 `gpt-4.1` 做真实验证**  
   - 这次终于拿到了真实模型结果，不再是鉴权错误、配额错误或保底兜底
   - 这说明项目已经跨过“接不通”的门槛，进入“能真实生成，但质量还要继续验证”的阶段

**这轮最大的认知更新**：
- **模型能力、平台配额、产品质量是三件不同的事**
- 以前容易把它们混在一起：一旦结果不好，就统称“模型不行”
- 但这次实际拆开后发现：
  - OpenAI 失败，主因是**账号额度**
  - FRIDAY `gpt-5.4` 失败，主因是**RPM 配额**
  - FRIDAY `gpt-4.1` 成功返回，但能否撑起产品价值，仍然要看**真实样本质量**

**为什么这件事对以后做 Vibe Coding 很重要**：
- 很多时候，团队以为“模型选型”只是技术实现细节，实际上它同时决定了：
  - 能不能在开发期反复试错
  - 能不能在首批用户验证阶段稳定跑起来
  - 会不会因为平台限流导致你误判产品方向
- 所以以后凡是做生成式产品，模型策略不能只写“用哪个模型”，还必须同时写清楚：
  - **开发联调模型**是谁
  - **正式生成模型**是谁
  - **降级模型**是谁
  - 它们各自的额度、RPM、价格、稳定性边界是什么

**本轮当前成果（到此为止真正成立的部分）**：
- 项目已经不再被单一模型绑定，具备了切换模型提供方的能力
- 已经验证了内部 FRIDAY 路径可用，`App ID + OpenAI 兼容接口` 方案成立
- 已经拿到 `gpt-4.1` 的真实返回结果，说明后续质量迭代终于可以建立在“真实强模型输出”上，而不是继续围着弱模型和 fallback 打转

**本轮仍未解决的问题（不能误判成已经成功）**：
1. **`gpt-5.4` 不是不能用，而是当前 App ID 暂时不够用**  
   这意味着“最强模型”并不一定等于“当前阶段最可用模型”。如果平台配额不够，再强也只是纸面能力。
2. **`gpt-4.1` 的单次成功不等于产品已经成立**  
   一条看上去还可以的结果，只能证明“有机会变好”，不能证明“已经稳定变好”。
3. **当前 prompt、校验器、重试策略大体还是旧骨架**  
   也就是说，这一轮的提升主要来自模型升级，而不是系统机制已经彻底进化。后续必须用样本压测来判断：旧 prompt 在更强模型上到底是“够用了”，还是“只是暂时没那么差了”。
4. **还没有形成固定 benchmark 集**  
   如果没有一组长期固定、跨度大的样本集，后面每一轮都可能被个别好结果或坏结果带偏。

**以后再遇到类似情况，建议直接按这个判断顺序走**：
1. 先判断是模型能力问题、平台额度问题，还是 prompt/校验问题
2. 不要把“接通了”误当成“成立了”
3. 不要把“一条看起来不错”误当成“已经稳定了”
4. 一旦强模型可用，下一步必须立即进入 benchmark，而不是继续凭感觉讨论

**经验教训**：
> 生成式产品里，"强模型接通"只是开始，不是结论。真正该庆祝的不是 API 200，也不是第一条看起来还行的结果，而是连续多组真实输入下，用户仍然觉得“这玩意真能连上”。
>
> 更具体一点：以后只要完成模型切换，下一步默认动作就应该是做 benchmark，而不是继续优化等待态、错误态或 fallback 文案。否则很容易再次把“工程进展”误判成“产品成立”。

### 4.8 第一轮真实样本压测结果（2026-05-07）

**为什么这一轮必须马上做**：
- 刚把模型切到 `FRIDAY + gpt-4.1`，如果不立刻做样本压测，团队很容易被“已经有一条成功结果”麻痹
- 这类产品最大的错觉就是：看到一条不错的路径，就以为核心惊喜已经成立；但真正决定产品生死的是**连续多组真实输入下的稳定命中率**
- 所以这轮压测的目标不是“证明已经成功”，而是尽快知道：现在距离“用户会愿意分享”还差多远

**压测样本（10 组，故意拉大跨度）**：
- `阿黛尔 -> AI Agent`
- `深海采矿 -> 韩国大选`
- `我想逃离996 -> 文艺复兴`
- `拼多多 -> 县城中学`
- `稳定币 -> 日本动漫`
- `外卖骑手 -> 自动驾驶`
- `比特币 -> 寺庙经济`
- `特朗普 -> 二次元`
- `宠物殡葬 -> 县域房价`
- `核聚变 -> 奶茶店`

**先记结论**：
- **工程层面明显进步了**：10 组里有 9 组拿到成功返回，说明 `gpt-4.1` 至少已经把“经常失败或直接掉 fallback”的问题压下去一大截
- **产品层面还远没到能放心宣布成立**：虽然 9/10 成功，但里面有相当多结果仍带明显的“编得通、但不够硬”的味道
- **当前最危险的新错觉** 不再是“总在保底”，而是“终于不用保底了，所以好像已经可以用了”

**这轮暴露出的真实情况**：
1. **主链路成功率上来了，但惊喜密度还没跟上**  
   比如 `深海采矿 -> 韩国大选`、`外卖骑手 -> 自动驾驶` 这类结果，已经出现了更像现实机制链的路径；但 `特朗普 -> 二次元`、`核聚变 -> 奶茶店` 这类结果，仍然能感觉到模型在“努力找一个能讲通的文化中介”，离“看完会想截图”还有距离。
2. **失败模式发生了变化：从“万能中介词”转成“貌似具体但未必扎实”**  
   以前最明显的坏结果是“技术进步 / 监管条款 / 行业标准”这种一眼假的万能桥。现在更常见的坏结果变成：
   - 节点看起来具体
   - 解释也像在说人话
   - 但你再追问一句“现实中真的是这样吗”，就会开始发虚
   这说明问题已经从“模型太弱”推进到了“模型会编 plausible story，但还不够硬”。
3. **校验器对“明显坏结果”能拦住一部分，但对“像真的但未必真”的内容无能为力**  
   这不是校验器失职，而是它的职责边界到了。像 `稳定币 -> 日本动漫` 这一组，最终被拦下，证明校验器对结构/硬约束仍有价值；但剩下通过的很多结果，本质上还是要靠人工评测才能识别“值不值得信”。
4. **当前的 benchmark 已经证明：产品进入了新的阶段**  
   现在不再是“先把模型换掉再说”，而是要开始解决更难的问题：
   - 什么样的路径算真的通过？
   - 什么样的路径虽然能过校验，但仍然不够妙？
   - 什么类型的输入最容易诱发模型编故事？

**如果只看这轮结果，最应该警惕的两个误判**：
- 误判 1：**9/10 成功返回 = 产品已经可用**  
  错。它只说明接口与模型底座明显改善，不说明用户会喜欢。
- 误判 2：**现在主要问题已经不是模型了**  
  也错。现在的问题不是“模型太弱”，而是“模型虽更强，但 prompt 和验收机制还没把强能力稳稳变成高质量产出”。

**这轮压测后，产品层最该做的事**：
1. **立刻把 benchmark 固化为回归集**：后面每次改 prompt、校验、重试，都必须复测这一组，不然很难知道是变好了还是只是换了一种翻车方式
2. **开始做人工质量分级，而不是只看成功/失败**：建议把结果分成 `明显可分享 / 勉强成立 / 结构成功但产品失败 / 直接失败` 四档
3. **把“看起来像真的但其实发虚”列为新的重点失败类型**：这一类比抽象万能桥更危险，因为它更容易骗过开发者自己
4. **优先提升高跨度词对的命中率**：像 `核聚变 -> 奶茶店`、`特朗普 -> 二次元` 这类样本，才最能证明产品的惊喜能力是否真的成立

**经验教训**：
> 生成式产品到中期，最危险的不是继续失败，而是开始“看起来像成功”。因为结果一旦变得更顺、更像人话，团队就更容易放松警惕。但用户真正买单的不是“像人话”，而是“这个连接真让我意外，而且我回头想想还真成立”。
>
> 以后只要主链路成功率显著提升，下一步就不要只问“成功率多少”，而要追问“这些成功里，有多少是真的能打动人”。这一步不做，产品还是会在更晚的时候翻车。

## 五、可复用的 Vibe Coding 原则（从本项目提炼）

1. **文档先行，但文档要能被执行**：写给 AI Agent 的文档不是给人欣赏的，每条都要有行动意义
2. **先零成本，再优质**：MVP 阶段用免费方案，调通逻辑再换更好的
3. **安全不是后期打磨项**：公开分享的产品从第一天就要正确处理凭证
4. **AI 的边界要显式定义**：AI 在哪做决策、在哪不该做决策，必须在产品层面想清楚
5. **传播机制是功能，不是运营**：单次闭环产品的分享能力要早做
6. **不做什么和做什么同等重要**：明确边界防止功能蔓延
7. **收口 ≠ 代码写完**：项目"完成"的验收标准应该是"本地 build 通过 + 服务器可跑 + 环境变量配好"，而不只是"代码落地"
8. **零成本优先有边界**：如果产品价值直接取决于生成质量，弱模型会比"没接模型"更伤产品；一旦进入真实用户验证阶段，就要果断把模型质量提到 P0
9. **模型可得性也是产品约束**：不要只选“理论上最强”的模型，还要同时考虑配额、RPM、预算、稳定性与切换成本
10. **强模型接通后先做 benchmark**：没有 benchmark，任何"感觉变好了"的判断都不可靠
11. **进入调优期前先整理结构**：Prompt 埋在臃肿文件里，调优效率会大打折扣，而且容易改乱

### 4.9 产品调优前的结构整理（2026-05-07）

**为什么要在调优之前先整理代码结构**：

压测结束之后，本打算直接开始改 Prompt，但发现 `api/entangle.ts` 已经膨胀到 769 行，Prompt 文本夹在校验逻辑和 fallback 数据之间。改一行 Good Case，需要先在 700 行里找到它在哪。

这带来了两个具体问题：
1. **调优效率低**：改 Prompt 要先花时间定位，容易因为看到附近的代码被分心，改着改着跑偏
2. **验证困难**：`local-server.mjs` 和 `api/entangle.ts` 的逻辑重复，改一边容易忘另一边，本地测试通过了但实际逻辑没改到

**这次整理做了什么（从产品视角看）**：

把代码按"谁负责什么"做了清晰分工：

| 职责 | 谁管 | 类比 |
|------|------|------|
| **"说什么给模型听"** | `api/_lib/prompt/entangle.ts` | 产品策划写的 Brief |
| **"什么结果算合格"** | `api/_lib/validate.ts` | 质检员的验收单 |
| **"不合格时怎么兜底"** | `api/_lib/fallback.ts` | 应急预案 |
| **"HTTP 请求怎么走"** | `api/entangle.ts` | 前台收单员 |

**以后调优时的操作流程变成了**：
- 改 Prompt、加好例/坏例 → 只开 `api/_lib/prompt/entangle.ts`
- 改质量门槛（什么算"泛"）→ 只开 `api/_lib/validate.ts`
- 改应急保底逻辑 → 只开 `api/_lib/fallback.ts`
- 每次改完之后同步更新 `local-server.mjs` 对应区块

**这次整理的产品教训**：

> 结构混乱不是工程师的问题，是产品节奏管理的问题。快速迭代期确实不需要"完美架构"，但一旦要开始"反复打磨某一层"（比如反复改 Prompt），就必须先把那一层独立出来。否则你以为在精雕，实际上是在乱砍。
>
> 判断"什么时候该整理结构"的简单原则：**如果你发现自己要改一个东西，但必须先搞清楚另外两个东西才敢动手，就说明结构已经妨碍调优了，应该先整理再继续。**

### 4.10 第一轮 benchmark 人工分级补录（A / B / C）

**分级标准先说清楚**：
- `A`：会让人想截图，回头追问也基本站得住
- `B`：能讲通，但还需要人工点头，不敢直接当产品惊喜
- `C`：结构成功但产品失败，或虽然返回了结果但本质上仍不够硬

| 样本 | 档位 | 一句话判断 |
|------|------|------------|
| `阿黛尔 -> AI Agent` | `B` | 方向是对的，但“巡演直播平台 / AI 数字分身公司”这类节点仍有较强编排感。 |
| `深海采矿 -> 韩国大选` | `A` | 能落到矿权、产业安全、就业情绪这类现实机制链，意外但回头能讲通。 |
| `我想逃离996 -> 文艺复兴` | `C` | 更像人生转译和审美跳跃，不是现实制度链或供给链。 |
| `拼多多 -> 县城中学` | `B` | 县域教育资源和物资流向有现实感，但部分组织节点仍像模型补出来的。 |
| `稳定币 -> 日本动漫` | `C` | 这一组最终被系统拦下，说明模型当时没找到足够硬的通道。 |
| `外卖骑手 -> 自动驾驶` | `A` | 从路测数据、标注外包到训练闭环，因果最清楚，是当时最像“真链路”的样本。 |
| `比特币 -> 寺庙经济` | `B` | 加密捐赠方向可讲通，但惊喜度和约束强度都还不够高。 |
| `特朗普 -> 二次元` | `C` | 明显偏向文化转译型编故事，不够硬。 |
| `宠物殡葬 -> 县域房价` | `B` | 土地、园区、商住开发这条线有现实抓手，但中间因果仍需要人工审核。 |
| `核聚变 -> 奶茶店` | `C` | 具体词变多了，但整体仍像为了连上而连上。 |

**这一轮的真实结论**：第一轮并不是“9/10 成功，所以产品可用”，而是 `A/B/C = 2 / 4 / 4`。也就是说，工程层的通路明显好转了，但真正能让人拍桌子的结果仍然很少。

### 4.11 Prompt 收紧后的第二轮复测（2026-05-07）

这次没有继续折腾 provider，而是直接改 Prompt，把模型从“能讲通就行”往“必须先走硬连接”推：
- 强制模型优先想 **供应链 / 合同 / 许可 / 预算 / 土地 / 基础设施 / 分发** 这类硬中介
- 要求每条路径里至少有 **2 个硬连接节点**
- 明确禁止“文化转译、情绪投射、圈层梗、现编机构名”这种看似具体、实际发虚的偷懒路线

**第二轮复测结果**：
- 10 组里 **10 组返回 200**
- 耗时约 **10.8s ~ 20.9s**
- 最明显的变化不是“突然更惊艳”，而是**路径骨架明显更硬了**：合同、采购、预算、土地、电网、物流、结算这类词开始系统性出现

| 样本 | 新档位 | 一句话判断 |
|------|--------|------------|
| `阿黛尔 -> AI Agent` | `B` | 从“数字分身公司”收敛到“场馆合同 -> 采集设备 -> 数据标注 -> 训练平台”，硬度明显提升，但还没到拍桌级。 |
| `深海采矿 -> 韩国大选` | `A` | “环保审批 -> NGO 诉讼与抗议 -> 青年政策动员”已经是可讨论的现实政治链。 |
| `我想逃离996 -> 文艺复兴` | `C` | 虽然补上了土地租赁、基金预算等硬节点，但整体仍是人生转译，不够自然。 |
| `拼多多 -> 县城中学` | `A` | “农产品供应链 -> 县域物流 -> 政府采购 -> 中学食堂/物资”是这轮最扎实的一条。 |
| `稳定币 -> 日本动漫` | `B` | 至少落到制作委员会、采购合同、结算平台，比上一轮直接失败前进了一步。 |
| `外卖骑手 -> 自动驾驶` | `B` | 物流园区、运营许可、调度平台是硬链，但“骑手”这个起点锚点感比上一轮略弱。 |
| `比特币 -> 寺庙经济` | `B` | “捐赠平台 -> 合规兑换 -> 运营预算”是一条清楚的落地链。 |
| `特朗普 -> 二次元` | `C` | “竞选资金募集 -> 对冲基金内容投资 -> 动画版权采购预算”仍有明显 plausible fiction 风险。 |
| `宠物殡葬 -> 县域房价` | `B` | 园区土地审批、财政预算、土地回流这几个节点让县域机制感更强了。 |
| `核聚变 -> 奶茶店` | `C` | 电网、租赁合同比上一轮硬，但仍偏顶层，离“妙”还有距离。 |

**第二轮最重要的判断**：
1. `C` 档确实减少了，说明 Prompt 收紧是有效的
2. 但 `A` 档没有爆发式增加，说明**我们解决的是“别那么虚”，还没解决“真惊喜”**
3. 这很像一个真实产品进入中期时该出现的状态：先把骨架拉正，再继续追求 Aha Moment

**对下一步的产品判断**：
- 现在最该做的，不是继续换模型，而是继续围绕 `C` 档样本做 Prompt 反例增强
- 第一优先盯住 `我想逃离996 -> 文艺复兴`、`特朗普 -> 二次元`、`核聚变 -> 奶茶店`
- 只要这三类高跨度输入还容易滑向“可讲通但不够妙”，产品就还没真正站稳

**这一轮新的经验教训**：
> Prompt 收紧的第一价值，不是让结果马上惊艳，而是先把“软联想”挤出去，让模型学会走硬骨架。真正的惊喜往往要建立在这一步之后，否则只是把更会讲故事误判成更懂现实。

> 更具体一点：以后每次看到新结果，先别问“像不像人写的”，先问“这条链里有没有合同、预算、土地、许可、物流、基础设施这类硬东西在流动”。这个问题答不上来，路径大概率还不够硬。

### 5.8 prompt是模型产品的灵魂
要给出goodcase和badcase，软硬约束结合
-虽然说有时候硬约束也挺好的，但是最好是大类，而非小类
prompt要有结构，按 System Prompt（一次配置，永久生效） 和 User Prompt（每次请求拼接） 两层来切
稳定不变 → System	角色、规则、定义、示例从不随用户输入变化
每次都变 → User	输入词、参数、上轮历史
