## 一、项目初始化与基础架构

### 1.1 本阶段目标

- 建立 React + Vite + TypeScript 可运行项目骨架
- 在没有现成 Node 环境的情况下，保证工程文件先落地
- 为后续 Serverless API 接入准备目录结构

### 1.2 实际过程与踩坑

**问题1：本机无 Node / npm / npx，初始化命令不可用**
- 现象：`npm create vite`、`npx create-vite` 直接报 `command not found`
- 影响：无法使用标准脚手架一键初始化

**问题2：环境安装链路不稳定**
- 尝试安装 nvm（GitHub/Gitee）出现超时
- `~/.nvm` 目录一度为空目录，导致 nvm 脚本无法加载
- 结论：在当前网络条件下，依赖在线安装环境会显著拖慢开发

**问题3：TypeScript 编译错误阻塞 build**
- 现象：`src/components/PathReveal.tsx` 中 `setSelectedNodeIndex` 的回调 `prev === sourceIndex ? null : sourceIndex` 因 `sourceIndex` 来自函数作用域，TS 推断返回类型不匹配
- 修复：将 `node.sourceIndex` 提取为本地 `const sourceIndex = node.sourceIndex`，再传入 setter，使类型推断稳定
- 教训：TS 对 setter 回调的返回类型推断比较严格，复杂的三元表达式里直接引用外部变量容易触发窄化失败

### 1.3 最终采取的方案

采取**手动骨架落地 + 事后补装环境**的两步策略：
1. 先不等环境，把项目结构与核心文件写齐（可离线推进的部分）
2. 收口阶段通过 nvm tarball 方式安装 Node.js，绕过 Git clone 网络问题

已落地文件：
- 基础工程：`package.json`、`index.html`、`vite.config.ts`
- TS 配置：`tsconfig.json`、`tsconfig.app.json`、`tsconfig.node.json`
- 前端目录：`src/`、`src/services/`、`src/types/`
- Serverless 目录：`api/`、`api/_lib/`
- 其他：`.gitignore`、`.env.example`

**环境最终状态**：Node.js v24.15.0（via nvm），npm 11.12.1，依赖已安装，`npm run build` ✅

### 1.4 经验教训

1. **Vibe Coding 的第一原则是不中断流**：环境卡住时，不要停工等环境，先把可离线推进的工程骨架落地。
2. **脚手架不是必须品**：`create-vite` 方便，但它产出的文件本质可手动还原。
3. **目录先行很重要**：先把 `src` 与 `api` 边界划清，后续迭代会更稳。
4. **nvm 网络问题的绕过方案**：遇到 GitHub/Gitee 超时时，改用 tarball 安装（下载 `.tar.gz` 解压后手动写入 `~/.bashrc`），比反复重试 git clone 快很多。
5. **build 通过是收口标准，不是可选项**：TS 编译错误可以在开发期被忽略，但 `tsc -b && vite build` 必须通过，否则无法部署。

## 二、Serverless 代理与 API 鉴权

### 2.1 本阶段目标

- 前端禁止直连模型 API
- 所有 AI 请求必须走 Serverless 网关
- 统一错误结构、请求 ID、基础限流

### 2.2 已实现内容

#### 2.2.1 网关工具层

文件：`api/_lib/llm.ts`

已实现：
- `buildRequestId()`：统一请求追踪 ID
- `jsonResponse()`：统一 JSON 响应 + CORS 头
- `createErrorPayload()`：统一错误结构（`code/message/retryable`）
- `parseJsonBody()`：请求体解析兼容
- `checkRateLimit()`：内存级限流（按 IP + 窗口）

#### 2.2.2 智谱 GLM JWT 鉴权

同文件实现了：
- `createZhipuToken(apiKey)`
- 使用 `apiKey` 拆分 `id.secret`
- HMAC-SHA256 签名生成短效 token（`exp`）
- `callZhipuChat()` 统一调用 `glm-4-flash`

### 2.3 踩坑与结论

1. **签名逻辑必须服务端处理**：前端做 JWT 生成会导致 Key 泄露风险。
2. **超时控制必须内建**：`fetch` 需配 `AbortController`，否则上游卡死会拖垮前端体验。
3. **统一错误结构非常关键**：前端只处理一个错误协议，能显著减少分支复杂度。

## 三、大模型调用与 JSON 解析

### 3.1 本阶段目标

- 实现 `POST /api/entangle` 与 `POST /api/daily-pair`
- 保证模型返回可被结构化解析
- 对非 JSON 输出提供容错

### 3.2 已实现内容

#### 3.2.1 关联路径接口

文件：`api/entangle.ts`

已实现：
- 参数校验：`termA`、`termB` 必填
- Prompt 约束：3-5节点、禁止万能中介、只输出 JSON
- 结构校验：`paths` 数量、`nodes` 数量、字段完整性
- 禁用中介词检查：技术进步/全球化/社会变迁/人类文明/时代发展
- 错误码映射：`E_BAD_REQUEST` / `E_MODEL_TIMEOUT` / `E_INVALID_MODEL_OUTPUT` / `E_PROVIDER_UNAVAILABLE` / `E_INTERNAL`

#### 3.2.2 每日配对接口

文件：`api/daily-pair.ts`

已实现：
- 要求模型生成真实可查话题，不得编造
- 解析失败时使用 `fallbackTopics`（真实词库）兜底
- 随机选取跨领域两词，符合“方案B：每次打开随机”

#### 3.2.3 JSON 容错策略

工具：`extractJson(text)`

顺序：
1. 优先提取 ```json 代码块
2. 失败后尝试提取 `{...}`
3. 再失败尝试提取 `[...]`
4. 全部失败抛错，返回 `E_INVALID_MODEL_OUTPUT`

### 3.3 经验教训

1. **“模型可控”是错觉，结构校验是必须**。
2. **先提取再校验** 比单次 `JSON.parse` 稳定得多。
3. **给每日配对加固定 fallback**，能保证首页永不空白。

### 4.4 为什么没有使用 React Flow

本阶段选择了更轻量的自定义轨道可视化而非 React Flow，原因：

1. 当前路径是严格线性结构（A → ... → B），不需要复杂画布能力

后续若要支持节点拖拽、自由缩放、多分支路径，再切换到 React Flow 更合适。

### 5.8 结果缓存与失败降级补齐（2026-05-06）

在明确历史记录不继续推进后，本轮下一步改为补齐真实可用性，而不是继续做回看功能。

落地位置：`src/App.tsx`

本轮新增能力：

- 为纠缠结果增加前端本地缓存（`localStorage`），缓存 key 基于词对归一化生成
- 缓存上限 12 条，缓存有效期 7 天，读取前会自动清理过期与无效结果
- 用户再次输入同一组词时，优先直接打开最近一次成功生成的路径，减少重复等待
- 当真实请求失败但本地有历史成功结果时，自动回退到缓存结果，不让主流程直接断掉
- 新增 `statusNote` 状态，用于向用户说明“当前展示的是缓存结果”而不是把它误报成错误

为什么这一步比历史记录更优先：

- 它直接服务当前主链路的稳定性，而不是新增一个旁支页面能力
- 它能收口“真实 API 不稳”与“示例预览态过强”之间的体验断层
- 它符合 README 当前交接说明里“分享卡片已落地 / 历史记录不做，优先收口真实产品化”的阶段目标

验收方式：

- 首次成功生成后，再次输入同词对可直接打开缓存结果
- 请求失败时若存在缓存结果，会回退显示并给出状态说明
- 若无缓存结果，则保持原有错误提示与重试逻辑不变

### 5.9 弱路径提示补齐（2026-05-06）

继续沿“非历史记录”的方向，本轮增加了对真实输出质量的前端提示收口。

落地位置：`src/App.tsx`

新增内容：

- 新增 `GENERIC_BRIDGE_TERMS`，用于识别常见空泛桥接词（如“技术进步”“全球化”等）
- 新增 `assessResultQuality(result)`，在结果可用后做轻量质量评估
- 当检测到“空泛桥接词”或“意外指数整体偏低”时，不阻断展示，但通过 `statusNote` 给出明确提示
- 提示文案会引导用户点击“换一条更妙的路径”，减少“看完觉得弱但不知道下一步怎么做”的断层

设计原则：

- 不替代后端校验，不改变既有 UI 结构
- 只补“可解释提示”，不做强制拦截，避免打断当前体验节奏

### 5.10 模型成功率与内容质量复盘（2026-05-07）

**触发现象**：
- 用户用“阿黛尔 → AI Agent”测试时，多次返回“监管条款 → 支付通道 → 行业标准”这类路径
- 用户明确判断：节点关系牵强、文案没深度，而且能看出是保底路径
- 这意味着系统虽然“稳定返回了结构化结果”，但主链路在产品层面已经失败

**先把问题分型，否则很容易瞎改**：
1. **结构失败**：模型没按 JSON 返回，或者字段不全，导致接口不能用
2. **语义失败**：JSON 是合法的，但中间节点空泛、连接词没因果、内容像模板拼接
3. **产品失败**：结构和语义勉强过关，但用户仍然觉得"不妙、不想分享、不值得看完"

这次的主要问题不是第 1 类，而是**第 2 类和第 3 类长期存在，却被 fallback 掩盖了**。

**根因诊断**：
1. **模型选型已经越过可用边界**  
   早期选择 `glm-4-flash` 是出于零成本与低延迟考虑，这在 MVP 搭骨架阶段成立；但在“从两个词生成 1-3 条有意外感、且严格符合 JSON 结构的路径”这个任务上，它的推理与结构遵从能力明显不足。问题不是偶发超时，而是模型能力本身不够支撑产品核心体验。
2. **Prompt 负载过重，仍按单轮一把梭处理**  
   当前 prompt 同时要求：理解产品定位、构造多条候选路线、筛掉通用桥接词、保证节点具体、写 `connection/detail/hook/summary`、并只输出 JSON。任务密度太高，但没有做分阶段生成，也只有一个正例，缺少反例约束，导致模型容易退化成“泛化词 + 安全写法”。
3. **系统优化方向一度偏向“让失败更像成功”**  
   之前对 `buildFallbackPaths()`、连接词模板和保底文案做了较多调整，确实能减少“看起来像兜底”的突兀感，但无法提升真实模型输出的相关度。只要主链路成功率低，fallback 再自然，本质上仍是在掩盖生成失败。
4. **重试策略并没有真正提供第二次机会**  
   两次请求本质上仍是同类 prompt、同一弱模型，第二次超时还更短；这更像“再赌一次”而不是“换一种更可能成功的生成策略”。因此重试对质量提升帮助有限。
5. **校验与降级策略优先保证结构稳定，而不是语义质量**  
   当前流程擅长识别“是不是合法 JSON”，不擅长提升“这条路径是不是妙”。一旦输出略有结构问题，就很快滑向 fallback；而 fallback 的存在又掩盖了主链路实际成功率。

**这类问题以后应该怎么排查，避免继续瞎调**：

**第一步：先量化主链路，而不是先看 fallback 好不好看**
- 记录 `primary_success_rate`：真实模型输出并通过校验、且不是 fallback 的比例
- 记录 `fallback_rate`：最终走到 `buildFallbackPaths()` 的比例
- 记录 `quality_pass_rate`：人工抽样后，真正被判断为"相关、具体、有意外感"的比例
- 如果 `fallback_rate > 30%`，说明主链路已经不稳定，不应该继续优先做前端体验层功能
- 如果 `quality_pass_rate < 50%`，说明即使接口能用，产品也还没成立

**第二步：每次只定位一层问题，不要多层同时乱改**
- 如果大量失败是 `cannot extract json` / `invalid response shape`，先改模型或输出约束
- 如果 JSON 合法但内容很泛，先改模型能力和 prompt 结构，不要先改前端提示
- 如果内容有一定质量但经常被误杀，再回头改校验逻辑
- 如果主链路成功率仍然低，禁止优先去润色 fallback 文案，因为那只会制造“好像解决了”的错觉

**第三步：针对这类多跳生成任务，优先按下面顺序优化**
1. **先升模型**：轻模型不适合承担“多跳联想 + 严格结构输出 + 高意外感”三重任务
2. **再拆 prompt**：不要把“发散构思、筛选路线、格式化输出”塞进一次回答里
3. **补反例而不只是补正例**：告诉模型什么叫坏路径，比只给一个好例子更能减少泛化输出
4. **把重试做成策略切换，不是重复抽卡**：第二次请求应该改变模型、温度、提示策略或生成步骤，而不是同样地再来一次
5. **最后才动 fallback**：fallback 是稳定性保险丝，不是效果增强器

**以后写生成式接口时，建议直接带上这份验收清单**：
- 连续测 10 组跨度大的真实输入，不要只测开发者自己最熟的词对
- 每组结果按 4 个维度打分：
  - `相关度`：中间节点是否真的连接起点与终点
  - `具体度`：是否落在具体对象、制度、作品、人物、产业，而不是空泛大词
  - `跳跃感`：是否有反常识但回头成立的一跳
  - `可传播性`：看到后是否值得截图或复述给别人
- 只有当这 4 个维度里至少 3 项稳定达标，才能认为主链路可用
- 如果评测时自己已经能一眼看出“又是模板词”，不要再骗自己说“用户也许能接受”

**这次暴露出来的几个工程反模式**：
- 用轻量模型承接高认知负载任务
- 同一个 prompt 同时承担创意生成、筛选和格式化三件事
- 把“返回合法 JSON”误写成主要验收标准
- 用更自然的 fallback 文案掩盖主链路失败
- 在没有主链路质量数据时，就开始做缓存、前端提示、包装性优化

**本轮结论**：
- `fallback` 仍然要保留，但必须退到最低优先级，只承担“页面不空白”的职责
- P0 目标应改为：**提高真实模型生成相关内容的成功率与质量**
- 后续优化顺序应是：
  1. 升级模型能力（优先从 `glm-4-flash` 升到更强模型）
  2. 重写 prompt 结构（分阶段、加反例、减少单轮负载）
  3. 重新设计重试策略（不是重复请求同一方案）
  4. 保留必要结构校验，但减少对优质结果的误杀
  5. 最后再看 fallback 触发文案，而不是反过来

**经验教训**：
> 对生成式产品来说，“接口稳定返回 JSON”不等于“功能可用”。如果用户持续看到的是 fallback，就说明真正的主流程已经坏了。工程上最危险的错觉，是把“稳定兜底”误当成“稳定可用”。
> 
> 更落地一点：以后凡是做生成式产品，都要单独维护一套“主链路命中率”指标和人工抽样评测，不要只盯日志里的 200、超时率和 JSON 解析成功率。那些只能证明系统活着，不能证明产品成立。

### 5.11 这次问题的技术诊断拆解（给未来排查时直接复用）

为了避免下次一上来就同时乱改模型、prompt、校验、超时、fallback，先把这次问题拆成可复用的诊断框架。

#### 5.11.1 先判断失败发生在哪一层

| 层级 | 典型现象 | 这次是否命中 | 第一优先动作 |
|------|----------|--------------|--------------|
| **接口层失败** | 超时、401、500、请求体错误 | 部分命中，但不是主因 | 修配置、鉴权、超时、网络 |
| **结构层失败** | 非 JSON、字段缺失、shape 不合法 | 偶发命中 | 优化输出约束与解析容错 |
| **语义层失败** | 节点空泛、关系牵强、像模板 | **主因** | 升模型、重写 prompt、调整重试策略 |
| **产品层失败** | 用户觉得无聊、不惊喜、不愿分享 | **主因** | 回到 Aha Moment 和样本评测 |

**为什么这张表重要**：
- 如果已经确认主要是语义层失败，就不要继续把精力投在 HTTP 错误包装、前端提示细节、fallback 文案自然度上
- 如果主因不是接口层，再继续做网络与结构稳定性优化，只会让失败更稳定，不会让结果更好

#### 5.11.2 这次 prompt 为什么实际执行效果差

当前任务本质上要求模型一次完成 5 件事：
1. 理解两个输入词背后的真实世界场景
2. 发散出多条潜在线路
3. 从中筛掉普通、空泛、套路化路径
4. 为每一跳补上因果解释与节点 detail
5. 最后严格收敛为固定 JSON

**对弱模型来说，这是典型的“任务堆叠过载”**。它最容易出现两种退化：
- 为了满足 JSON 约束，牺牲内容质量，输出最安全的抽象桥接词
- 为了满足“看起来有逻辑”，拼出一条表面顺滑、实则无信息增量的模板路径

**以后遇到类似任务，prompt 至少要遵循这 6 条**：
1. **分阶段**：发散构思、筛选路线、格式化输出，最好不要完全压在同一轮里
2. **正反例都要给**：只给好例子，模型会模仿形式；给坏例子，模型才知道什么不能做
3. **反例要贴近真实失败模式**：比如“技术进步→平台生态→行业标准”这类词串，就应该直接写进 bad case
4. **规则按优先级排序**：最重要的约束放前面，比如“节点必须具体”和“不得偷换输入词”应先于 JSON schema
5. **不要把产品介绍写得太抒情**：模型会被世界观文字分散注意力，忽略真正的执行规则
6. **要求可执行，不要求空洞审美词**：像“更妙”“更高级”这种词必须被翻译成具体判据，否则模型无法稳定遵循

#### 5.11.3 校验逻辑该保什么、该让什么

这次一个很重要的认知是：**校验器不是内容创作者，它只能做守门员**。

**校验器必须继续做的事**：
- 结构合法性：能 parse、字段齐、数组长度合理
- 基础约束：节点不重复、不等于起终点、条数范围正确
- 明显坏词拦截：`技术进步`、`全球化`、`社会变迁` 等黑名单词
- 低质量连接词拦截：`A 导致 B` 但没有任何具体机制、或只是重复节点名改写

**校验器不该妄图做的事**：
- 用简单字符串规则去判断复杂语义是否“真的相关”
- 因为一两处措辞不理想就直接误杀整条可能不错的路径
- 为了提升“看起来稳定”，过早把结果送进 fallback

**更适合的校验思路**：
- **硬拦截**：结构错、黑名单词、明显空泛模板词，直接判失败
- **软评分**：对 `connectionToNext`、`detail`、节点具体度做轻量打分
- **阈值决策**：只有当总分明显过低时才重试；边缘结果优先重试模型，不要马上 fallback

也就是说，校验应该做成“拦明显坏结果 + 给重试提供信号”，而不是“替模型下内容判断”。

#### 5.11.4 超时、重试、fallback 应该怎么排优先级

这次最大的策略偏差之一，是让 fallback 太容易接管主流程。后续可以直接按下面的顺序处理：

**1. 超时策略**
- 高认知负载任务，不要用为轻接口设计的短超时
- 如果模型需要构思、筛选、再格式化，20-30 秒超时比 8-10 秒更合理
- 第二次重试不应更短，除非它改成了更轻的任务；否则只是更快失败

**2. 重试策略**
- 重试必须代表“换打法”，不是“再抽一次同样的卡”
- 可切换项包括：
  - 模型（轻模型 → 强模型）
  - prompt 结构（一步式 → 分阶段式）
  - 约束重心（先保具体度，再保文案）
  - 温度与采样策略
- 如果两次请求模型、prompt、采样都差不多，就不该把它叫“第二套策略”

**3. fallback 策略**
- fallback 的职责只能是：页面不空白、链路不断、用户知道可以再试
- fallback 不应该承担“看起来也挺像真结果”的职责，否则团队会失去对主链路健康度的敏感性
- 更稳妥的做法不是把 fallback 做得更真，而是让系统更愿意继续争取真实产出

#### 5.11.5 下次改动前，先回答这 10 个技术问题

1. 当前主要失败是接口层、结构层、语义层还是产品层？
2. 当前主链路成功率是多少？fallback 触发率是多少？
3. 失败样本里，最常见的坏节点词有哪些？是否该写入黑名单？
4. prompt 是否同时承担了“构思 + 筛选 + 格式化”三件事？
5. 第二次重试和第一次相比，究竟改变了什么？
6. 当前超时阈值，是按真实任务负载设的，还是拍脑袋设的？
7. 校验器是在拦明显坏结果，还是在替模型做内容评判？
8. fallback 的存在，是不是已经让团队忽略了主链路成功率？
9. 当前改动能否直接提升主链路质量？如果不能，优先级是否该下调？
10. 有没有用 10 组跨度大的真实输入做过人工抽样，而不只是测开发者自己熟悉的几组词？

**结论**：
以后这类生成式接口的技术路线，不应再是“先保结构稳定，再慢慢补内容”，而应该改成：
- 先确认模型能力是否跨过任务门槛
- 再把 prompt 做成可执行的策略，而不是大段说明书
- 再让校验与重试帮助主链路，而不是过早触发 fallback
- 最后才补那些让失败更平滑的工程细节

### 5.12 模型提供方升级与 Friday 接入验证（2026-05-07）

在把“提高主链路质量”确认为 P0 之后，本轮没有继续打磨 fallback，而是直接推进模型底座升级与真实联调。

**本轮技术目标**：
1. 让项目不再只绑定单一模型提供方
2. 验证内部 FRIDAY One-API 是否能作为新的主模型入口
3. 区分“接不通”“额度不够”“模型质量不够”三类问题，避免继续混着改

**本轮实际改动**：

#### 5.12.1 调用层从“单提供方”改为“多提供方优先级路由”

落地文件：`api/_lib/llm.ts`、`local-server.mjs`

已落地能力：
- 新增 `FRIDAY_BASE_URL = https://aigc.sankuai.com/v1/openai/native`
- 新增 `DEFAULT_FRIDAY_MODEL = gpt-5.4`
- 统一 provider 选择逻辑，优先级变为：
  1. `FRIDAY_APP_ID`
  2. `OPENAI_API_KEY`
  3. `ZHIPU_API_KEY`
- 继续保持 OpenAI 兼容请求格式：`POST {base}/chat/completions`
- `local-server.mjs` 启动时会打印当前实际命中的提供方与模型，方便本地联调时一眼确认配置是否生效

**为什么这一步重要**：
- 以前每次换模型，都像是在重做一遍接入；现在改成了“配置切换”，后续可以更低成本做 A/B 测试和临时降级
- 这让“模型选型”从一次性决定，变成可以被真实样本和平台额度反向校正的策略变量

#### 5.12.2 环境变量模板与本地凭证策略同步调整

落地文件：`.env.example`、`.env`

本轮处理：
- `.env.example` 改成以 `FRIDAY_APP_ID` 为主配置，明确写出 `FRIDAY_BASE_URL` 与 `FRIDAY_MODEL`
- 本地 `.env` 中移除了之前写入的 OpenAI 官方 key，避免继续混用外部额度与内部平台
- 错误文案同步升级，从“未配置 OPENAI/ZHIPU”改成“未配置 FRIDAY_APP_ID、OPENAI_API_KEY 或 ZHIPU_API_KEY”

**经验教训**：
- 一旦 provider 策略变了，模板和错误提示必须一起改；否则后续排错时很容易被旧文案带偏
- 删除已失效的敏感配置，比“先留着以后也许用”更安全；尤其是多人协作或长期迭代项目里，旧 key 常常会制造隐性歧义

#### 5.12.3 真实联调结果：这次不是一个问题，而是三类问题被依次拆开了

**阶段 A：OpenAI 官方 API 接入**
- 现象：调用链路打通，请求能发到 OpenAI
- 真实返回：`429 insufficient_quota`
- 结论：不是代码接入错误，而是账号/项目额度不足
- 技术价值：证明当前代码结构已经足够支撑多 provider，不必再把“换模型”视为大改

**阶段 B：FRIDAY One-API + gpt-5.4**
- 现象：`FRIDAY_APP_ID` 生效，本地服务明确显示当前 provider 为 `FRIDAY_APP_ID (gpt-5.4)`
- 真实返回：`429 App:**3041在模型:gpt-5.4每分钟请求次数超过限制`
- 结论：接口、鉴权、模型名都没问题，卡点在该 App ID 对 `gpt-5.4` 的 RPM 限流
- 技术价值：把问题从“是不是接错了”收敛为“平台配额是否足够”

**阶段 C：FRIDAY One-API + gpt-4.1**
- 现象：将 `.env` 中 `FRIDAY_MODEL` 从 `gpt-5.4` 切到 `gpt-4.1`
- 真实结果：`POST /api/entangle` 成功返回完整路径 JSON
- 结论：当前内部平台路径已具备真实可用性，项目已经跨过“接不通/打不通”的门槛
- 但也要明确：这只证明“上游成功返回”，还不能证明“产品级质量稳定成立”

#### 5.12.4 这轮迭代后的真实状态判断

**已经解决的问题**：
1. **模型接入不再被单一路径绑定**：外部 OpenAI、内部 Friday、智谱三路已经具备可切换能力
2. **Friday 接入方式已经验证成立**：`App ID + One-API OpenAI 兼容接口` 可用
3. **强于 `glm-4-flash` 的模型已跑通真实样本**：至少 `gpt-4.1` 已拿到结构完整结果

**尚未解决的问题**：
1. **上游错误被过度抽象**  
   当前前端/接口层对 `429 RPM exceeded`、`insufficient_quota` 这类问题仍统一映射为“上游服务暂不可用”，排查效率不够高，也容易让用户误以为是平台宕机。
2. **当前成功主要来自模型升级，而不是机制彻底升级**  
   prompt、重试、校验虽然前面已经强化，但这次的显著改善首先来自模型变强。换句话说，系统机制是否已经足够稳，还必须靠 benchmark 验证。
3. **还没有主链路质量数据**  
   目前只拿到单点真实样本，尚无 `gpt-4.1` 下的 `primary_success_rate / fallback_rate / quality_pass_rate`。
4. **命名债务开始显现**  
   当前内部函数名仍保留 `callZhipuChat()` / `callZhipu()` 这类历史命名，虽然功能上已支持多 provider，但后续继续迭代时会增加理解负担。

#### 5.12.5 这轮之后，进一步解决思路应该怎么排

**第一优先：做 benchmark，而不是继续凭感觉判断**
- 固定 8-10 组跨度大的真实词对
- 对每组记录：
  - 模型名
  - 是否命中真实主链路
  - 是否触发 fallback
  - 人工质量评分（相关度 / 具体度 / 跳跃感 / 可传播性）
- 没有这套数据前，不要宣布“gpt-4.1 已经够用”或“prompt 已经大幅变好”

**第二优先：把 provider 相关错误再细分一层**
- `429 + RPM exceeded` → 明确提示“当前模型分钟配额已满”
- `429 + insufficient_quota` → 明确提示“当前账号/项目额度不足”
- `401/403` → 明确提示鉴权失败
- 这样做的目的不是为了让提示更好看，而是为了防止后续继续把平台问题误判成 prompt 问题

**第三优先：继续把强模型用于“正式生成”，把便宜模型留给联调**
- 如果 `gpt-5.4` 后续 RPM 能申请到，可以考虑把它作为“正式生成模型”
- 当前阶段更务实的做法是：
  - `gpt-4.1` 作为真实内容验证模型
  - 其他低成本模型作为开发调试或降级模型
- 不要再次回到“为了免费先默认弱模型”的惯性里

**第四优先：后续再收命名与观测性债务**
- 把 `callZhipuChat` 这类命名改成 provider-neutral 的 `callChatCompletion`
- 在日志中附带 provider/model/requestId，方便后续 benchmark 和问题定位
- 但这些都应排在 benchmark 之后，因为它们提升的是调试效率，不是内容质量本身

**本轮经验教训**：
> 对生成式产品来说，模型升级不是一句“换更强的”就结束了。真正该拆开的至少有三层：代码是否支持切换、平台是否给得出额度、模型返回后用户是否觉得值。
>
> 更具体一点：以后只要做 provider 切换，都要顺手记录 3 件事——“接入是否成功”“配额是否足够”“质量是否稳定”。少记任何一层，后面都很容易把问题看错。

### 5.13 第一轮真实样本压测（FRIDAY `gpt-4.1`，2026-05-07）

在 `FRIDAY + gpt-4.1` 第一次成功返回真实路径之后，本轮立刻补了一次 10 组样本压测，目的是避免被单个好结果带偏判断。

#### 5.13.1 压测目的

这轮压测不是为了跑性能数据，而是为了拿到第一版**主链路质量基线**。重点回答四个问题：
1. 当前 `gpt-4.1` 下，真实主链路成功率大概是多少？
2. 失败主要发生在接口层、结构层，还是语义/校验层？
3. 哪类输入最容易出“貌似具体、实际发虚”的结果？
4. 现在更应该继续改模型选择，还是回头细化 prompt / 校验 / benchmark 机制？

#### 5.13.2 样本集与执行方式

执行方式：本地服务启动在独立端口，逐组请求 `POST /api/entangle`，请求间隔 1.2~1.5 秒，避免因为连发导致额外限流噪声。

本轮样本共 10 组：
1. `阿黛尔 -> AI Agent`
2. `深海采矿 -> 韩国大选`
3. `我想逃离996 -> 文艺复兴`
4. `拼多多 -> 县城中学`
5. `稳定币 -> 日本动漫`
6. `外卖骑手 -> 自动驾驶`
7. `比特币 -> 寺庙经济`
8. `特朗普 -> 二次元`
9. `宠物殡葬 -> 县域房价`
10. `核聚变 -> 奶茶店`

选择逻辑：
- 有人物 / 技术 / 政治 / 经济 / 县域 / 文化等不同域
- 同时包含“相对好连”的词对和“非常容易编故事”的词对
- 刻意混入高跨度样本，逼模型暴露真实上限，而不是只测容易题

#### 5.13.3 原始结果概览

**整体结果**：
- 10 组中 **9 组返回 200**
- 1 组返回 **502 `E_INVALID_MODEL_OUTPUT`**
- 耗时区间约 **12.76s ~ 33.91s**

**逐组结果摘要**：
- `阿黛尔 -> AI Agent`：200，约 12.83s，返回 2 条路径，结构完整；但存在“巡演直播平台 -> 表演行为数据库 -> AI数字分身公司”这类较强叙事性节点，真实性仍需人工审视
- `深海采矿 -> 韩国大选`：200，约 20.67s，返回 2 条路径；是本轮相对更接近“现实机制链”的样本之一
- `我想逃离996 -> 文艺复兴`：200，约 28.99s，返回 1 条路径；结构完整，但“民营画室 -> 素描教材 -> 美术考级 -> 博物馆展览”更像可讲通的人生转译路径，而不是强制度链
- `拼多多 -> 县城中学`：200，约 14.54s，返回 2 条路径；其中“奖学金供应链”路径有一定现实感，但“广告合作项目 -> 教育基金会 -> 评审委员会”仍需怀疑是否过于编排
- `稳定币 -> 日本动漫`：502，约 33.91s，错误为 `E_INVALID_MODEL_OUTPUT`；日志显示多轮尝试未通过质量校验，属于当前少数被系统明确拦下的失败样本
- `外卖骑手 -> 自动驾驶`：200，约 15.93s，返回 2 条路径；“路测数据采集 -> 标注外包 -> 自动驾驶训练”链条较有现实说服力，是本轮较好的正样本
- `比特币 -> 寺庙经济`：200，约 12.76s，返回 2 条路径；能看出模型开始用“加密捐赠平台 / NFT 项目”这类具体节点，但 `surpriseIndex` 偏低，且 plausibility 仍不算硬
- `特朗普 -> 二次元`：200，约 13.82s，返回 2 条路径；较明显暴露出“文化转译型编故事”倾向
- `宠物殡葬 -> 县域房价`：200，约 33.3s，返回 1 条路径；“集体建设用地 -> 园区开发 -> 商住开发 -> 房价动态”链条可讲通，但中间因果强度仍需人工审核
- `核聚变 -> 奶茶店`：200，约 14.22s，返回 2 条路径；是本轮“看似具体、但最容易发虚”的代表样本之一

#### 5.13.4 从日志看见的真实失败模式

压测日志里出现了这些关键失败信号：
- `Expected ',' or ']' after array element in JSON...`：说明即使在更强模型上，结构失败仍然存在，只是频率下降
- `node term length invalid`：说明校验器对节点长度等硬约束仍在发挥作用
- `E_INVALID_MODEL_OUTPUT`：说明至少有一部分“虽然返回了文本，但不值得放行”的结果被系统挡住了

这说明当前系统已经从“弱模型 + 高频保底”进入到新的阶段：
- **结构失败仍存在，但已不是主流矛盾**
- **更大的问题开始变成：通过结构与校验的结果里，有多少是真正高质量的**

#### 5.13.5 这轮压测得出的技术判断

**判断 1：当前主链路成功率显著上升，但不能把 9/10 直接等价为“可用率”**
- 9/10 只能说明：接口、模型能力、JSON 结构遵从度、超时策略都比之前明显改善
- 它不能说明：9 条里有多少真的值得给用户看
- 所以后续指标至少要拆成：
  - `response_success_rate`
  - `validation_pass_rate`
  - `human_quality_pass_rate`

**判断 2：失败重心已经从“抽象万能桥”部分转移到“貌似具体的 plausible fiction”**
- 以前最容易看出的坏结果是“技术进步 / 监管条款 / 行业标准”
- 现在更多结果会给出具体机构、公司、项目名，读起来更像真事
- 但这些具体名词并不自动等于高质量，因为模型可能只是把故事讲得更像样了
- 这类问题比抽象桥更危险，因为它更容易骗过开发者自己的第一眼

**判断 3：当前校验器已经不可能单靠字符串规则解决主要质量问题**
- 它对结构错误、长度错误、黑名单词仍然有价值
- 但面对“看似具体但未必成立”的路径，纯规则几乎无能为力
- 这进一步证明：后续必须建立 benchmark + 人工评分，不要试图把所有质量判断都塞回校验器

**判断 4：当前最值得继续提升的不是 provider 层，而是 prompt 与验收机制**
- provider 这轮已经证明能跑通
- 下一步再继续折腾接入层，收益会明显低于：
  1. 优化 prompt 的阶段拆分
  2. 收紧反例与现实锚点要求
  3. 建立固定 benchmark 回归集
  4. 对结果做人工分级打分

#### 5.13.6 进一步解决思路（基于本轮压测，而不是拍脑袋）

**第一优先：固定 benchmark，不允许后续再只凭感觉判断**
- 把这 10 组样本直接固化为第一版回归集
- 后续每轮改 prompt / 校验 / 重试 / 模型时都必须复测
- 否则很容易出现“某 2 组变好了，但另 3 组偷偷变差了”却没人察觉

**第二优先：给 benchmark 加人工分档，而不是只记 200/502**
- 建议分四档：
  - A：明显可分享
  - B：勉强成立
  - C：结构成功但产品失败
  - D：直接失败
- 没有这层人工分档，日志只能证明系统活着，不能证明产品成立

**第三优先：针对“plausible fiction”改 prompt，而不是继续堆黑名单**
- 增加“不要虚构机构/项目/平台，如不确定请用更通用但真实的制度或产业节点”
- 加强“每一跳必须是现实中可指认的媒介、制度、组织、资产或基础设施”
- 强迫模型优先走“供应链 / 法规 / 平台 / 组织 / 基础设施”这类硬中介，而不是文化类软转译

**第四优先：保留当前硬校验，但不要再高估它对产品质量的决定作用**
- 它能继续拦结构坏结果
- 但对产品价值更大的部分，必须依赖 benchmark 与人工评测

**本轮经验教训**：
> 生成式接口进入强模型阶段后，最容易犯的新错误不是“结果太假”，而是“结果假得更像真”。这会让团队从“能一眼看出不行”进入“需要仔细想一想才发现不够硬”，从而更容易误判产品已经成熟。
>
> 更落地一点：以后 benchmark 里要专门增加一类判据——"这条路径是不是只是讲得像，而不是现实里真的有足够硬的连接"。如果不专门盯这个问题，强模型时代照样会翻车。

### 5.14 代码结构整理（2026-05-07）

#### 5.14.1 整理动机

压测结束、进入产品调优阶段之前，发现现有代码结构存在严重的"胖文件"问题：
- `api/entangle.ts`（769 行）：Prompt 文本、结构校验、质量过滤、fallback 数据、错误映射全部堆在一起
- `local-server.mjs`（856 行）：大量逻辑与 `api/entangle.ts` 重复，但维护时难以发现
- 核心问题：**改 Prompt 需要在 700+ 行里找，改校验规则不知道是改 prompt 还是改校验，两者边界不清**

这种结构在功能迭代期就很麻烦，进入产品调优期（频繁修改 Prompt、调整质量门槛）会更严重。

#### 5.14.2 整理方案

将 `api/entangle.ts` 按职责拆分为四个模块：

| 文件 | 职责 | 调优时机 |
|------|------|----------|
| `api/_lib/prompt/entangle.ts` | Prompt 模板、Good/Bad Cases、多轮尝试配置 | 每次产品调优时 |
| `api/_lib/validate.ts` | 结构校验（硬性）、质量过滤（软性）、类型定义、错误分类 | 调整验收标准时 |
| `api/_lib/fallback.ts` | 保底节点数据、保底路径构建、fallback 触发条件 | 调整保底策略时 |
| `api/entangle.ts` | HTTP 路由、参数校验、错误响应映射（薄薄一层） | 改接口行为时 |

`local-server.mjs` 同步按同样的分层顺序重新整理，并在文件顶部标注与 TS 模块的对应关系，确保后续改任意一侧时都知道要同步哪里。

#### 5.14.3 拆分后的文件行数对比

| 文件 | 整理前 | 整理后 |
|------|--------|--------|
| `api/entangle.ts` | 769 行 | ~110 行 |
| `api/_lib/prompt/entangle.ts` | 不存在 | ~140 行 |
| `api/_lib/validate.ts` | 不存在 | ~200 行 |
| `api/_lib/fallback.ts` | 不存在 | ~120 行 |
| `api/_lib/llm.ts` | 213 行 | 213 行（不动） |
| `local-server.mjs` | 856 行 | ~580 行（结构更清晰） |

#### 5.14.4 这次整理的关键原则

1. **每个文件只回答一个问题**：prompt/ 只回答"发给模型什么"；validate 只回答"模型返回了什么算合格"；fallback 只回答"不合格时怎么保底"；entangle.ts 只回答"HTTP 请求怎么处理"。

2. **调优入口清晰**：以后改 Prompt、加 Good/Bad Case，只动 `api/_lib/prompt/entangle.ts`，不需要看其他任何文件。

3. **local-server.mjs 保持镜像关系**：由于不能直接 import TS 文件，local-server.mjs 仍然复制核心逻辑，但通过顶部注释的"模块对应表"明确要同步哪里，不再靠记忆。

#### 5.14.5 经验教训

> 结构整理不是锦上添花，而是产品调优的前置条件。如果 Prompt 埋在几百行代码中间，改一次需要找半天；如果改了 TS 文件忘了同步 mjs，本地测试结果和线上不一致，调优就会变成自欺欺人。
>
> 具体原则：**只要进入高频迭代阶段（每天改 Prompt 或校验规则），就先把文件职责拆清楚，再开始调。否则越改越乱，而且很难判断"是改对了还是改错了"**。

#### 5.15.2 Prompt 收紧策略

针对首轮暴露出的 `plausible fiction` 问题，本轮没有继续折腾 provider，而是直接收紧 `api/_lib/prompt/entangle.ts` 与 `local-server.mjs` 中的 Prompt 镜像：

1. **强制优先硬连接**
   - 在策略层明确优先 `供应链 / 合同 / 许可 / 牌照 / 预算 / 采购 / 土地 / 园区 / 物流 / 基础设施 / 诉讼 / 补贴`
   - 要求每条路径中至少有 `2` 个中间节点属于硬连接对象

2. **显式打击软连接偷懒路线**
   - 禁止只靠文化符号、审美趣味、社会情绪、圈层迁移过桥
   - 在 `single_path` 策略里明确删除“文化联想 / 情绪投射 / 圈层梗”成立的节点

3. **抑制现编机构名**
   - 如果用公司、项目、基金会、平台、委员会等专有名词，要求必须是现实中广为人知或高度确定存在的对象
   - 如果拿不准，就改写成更通用但真实的制度 / 产业类别

4. **强化机制表述要求**
   - `connectionToNext` 不再只要求“写具体机制”，而是明确资源、权力、资金、数据、货物、土地或人力如何传递

#### 5.15.5 本轮结论

1. **Prompt 收紧是有效的**
   - 最直接的变化不是更惊艳，而是骨架更硬
   - `C` 档从 `4` 组下降到 `3` 组
   - `稳定币 -> 日本动漫` 从首轮失败提升到 `B`

2. **当前解决的是“少发虚”，还不是“高惊喜”**
   - `A` 档仍然只有 `2` 组，没有出现爆发式提升
   - 说明这轮主要解决了 plausibility 下限，没有解决 Aha Moment 上限

3. **高跨度难题仍是下一轮核心战场**
   - `我想逃离996 -> 文艺复兴`
   - `特朗普 -> 二次元`
   - `核聚变 -> 奶茶店`

这 3 组仍然容易滑向“讲得通但不够妙”的结果，后续应围绕它们继续增加反例与更强约束。

#### 5.15.6 经验教训

> Prompt 收紧的第一价值，不是立刻把结果变成 A，而是先把“软联想”和“现编机构名”挤出去，让模型学会稳定走硬骨架。真正的高惊喜，通常要建立在这一步之后。
>
> 更落地一点：以后 benchmark 复盘时，不能只问“像不像人写的”，而要先问“合同、预算、许可、土地、物流、基础设施这些硬东西到底有没有在流动”。如果这个问题答不上来，路径大概率还不够硬。
